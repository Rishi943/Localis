## Accomplishments

- Added notes.retrieve execute_tool branch: queries notes table (with optional text filter), injects up to 20 notes into system prompt context
- Defined NOTES_TOOL_SCHEMAS module-level constant with full JSON schema for both tools (same format as assist.py's _build_tool_schema())

## Files Created/Modified
- `app/main.py` - ALLOWED_TOOLS expansion, execute_tool branches for notes.add/notes.retrieve, NOTES_TOOL_SCHEMAS constant

## Decisions Made
- notes.add branch accesses SQLite directly via `database.DB_NAME` rather than calling the `/notes/add` HTTP endpoint — avoids the FastAPI request cycle overhead in an async tool handler
- NOTES_TOOL_SCHEMAS is a module-level constant, not injected into any specific system prompt string — it documents the tool schemas for future LLM router integration (the current architecture is frontend-driven tool selection, not LLM-router-driven)
- notes.retrieve result uses `role: "system"` so it merges into the system prompt blob (same as memory_retrieve) — keeps notes out of conversation history to prevent hallucination from irrelevant prior note content

## Deviations from Plan

None - plan executed exactly as written.

The plan referenced a "router system prompt" which doesn't exist as a distinct string in the current codebase (tool selection is frontend-driven). The closest equivalent is the NOTES_TOOL_SCHEMAS module-level constant, which satisfies all acceptance criteria: notes.add and notes.retrieve appear in a prompt/schema string outside of execute_tool and ALLOWED_TOOLS, including note_type, due_at parameters and "remind me", "add note", "remember" language.
